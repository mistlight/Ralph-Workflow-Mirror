# Git Workflow Architecture

This document describes Ralph's git workflow architecture, including how commits are created, how agents are isolated from git operations, and how diffs are generated for reviewers.

## Overview

Ralph implements a sophisticated git workflow that creates commits automatically throughout the development process while keeping AI agents completely isolated from git operations. This ensures:

1. **Per-iteration commits**: Every development iteration creates a commit (if there are meaningful changes)
2. **Per-review-cycle commits**: Every review-fix cycle creates a commit (if fixes were applied)
3. **Agent isolation**: AI agents never receive git context or commands
4. **Cumulative diffs for reviewers**: Reviewers see all changes since the pipeline started

## Per-Iteration Commits

### Development Iterations

After each development iteration, Ralph automatically creates a commit:

**Location**: `src/phases/development.rs` (see commit handling logic)

```rust
// Check for changes and create commit if modified
let snap = git_snapshot()?;
if snap != prev_snap {
    ctx.logger.success("Repository modified");
    ctx.stats.changes_detected += 1;

    // Create a commit with auto-generated message
    if let Some(agent_cmd) = ctx.registry.developer_cmd(ctx.developer_agent) {
        match commit_with_auto_message_result(&agent_cmd) {
            CommitResult::Success(oid) => {
                ctx.logger.success(&format!("Commit created successfully: {}", oid));
                ctx.stats.commits_created += 1;
            }
            CommitResult::NoChanges => {
                ctx.logger.info("No commit created (no meaningful changes)");
            }
            CommitResult::Failed(err) => {
                ctx.logger.error(&format!("Failed to create commit: {}", err));
                return Err(anyhow::anyhow!(err));
            }
        }
    }
}
prev_snap = snap;
```

### Review-Fix Cycles

After each review-fix cycle, Ralph also creates a commit:

**Location**: `src/phases/review.rs` (see commit handling logic)

The same logic as development iterations is applied - changes are detected and a commit is created with an auto-generated message.

## Commit Message Generation

### How It Works

Commit messages are generated by the orchestrator (not agents) using this flow:

1. **Get the diff**: `git_diff()` returns HEAD → working tree diff
2. **Validate diff size**: Large diffs (>100KB) trigger warnings, very large diffs (>500KB) are truncated
3. **Check for meaningful changes**: `has_meaningful_changes()` filters whitespace-only
4. **Generate commit message**: LLM is called with an enhanced prompt that emphasizes analyzing code changes
5. **Fallback if LLM fails**: Generate an intelligent fallback message that identifies file types
6. **Stage all changes**: `git_add_all()` stages everything
7. **Create commit**: `git_commit()` creates the commit using libgit2

**Location**: `src/phases/commit.rs` (see `commit_with_generated_message`)

### Enhanced Prompt

The commit message generation prompt has been significantly improved to produce higher quality messages:

- **Analysis instructions**: LLM is explicitly instructed to analyze what changed and why
- **Type guidelines**: Detailed descriptions of each commit type (feat, fix, docs, etc.)
- **Good examples**: Multiple concrete examples of well-formatted commit messages
- **Bad examples**: Explicit examples of what to avoid (e.g., "chore: x files changed")
- **Emphasis on semantic meaning**: Focus on what changed functionally, not file counts

**Location**: `src/prompts/commit.rs` (see `prompt_generate_commit_message_with_diff`)

### Diff Size Validation

Ralph uses adaptive diff size management to prevent issues with large diffs:

- **Agent-specific limits**: Different agents have different token limits (GLM: ~100KB, Claude: ~300KB)
- **Progressive truncation**: Prompts are tried with full diff first, then progressively truncated
- **Context preservation**: When truncating, context around changes is preserved

**Location**: `src/phases/commit.rs` (see `get_max_prompt_size_for_agent`)

### Intelligent Fallback Messages

When LLM commit message generation fails, the fallback system uses a multi-stage approach:

1. **Prompt variants**: Try up to 8 different prompt formats with progressive truncation
2. **Agent chain**: Try all configured commit agents in sequence
3. **Emergency prompts**: Try minimal prompts without diff content
4. **Hardcoded fallback**: As absolute last resort, use `chore: automated commit`

The staged prompt truncation system adjusts diff size progressively to find a working prompt size for each agent. This ensures commits succeed even with very large diffs.

**Location**: `src/phases/commit.rs` (see `generate_commit_message_with_fallback_chain`)

### Commit Message Format

Commit messages follow the **Conventional Commits** format:

```
<type>[optional scope][!]: <subject>

[optional body]

[optional footer]
```

**Types**: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `build`, `ci`

**Example**:
```
feat(auth): add OAuth2 login flow

Implement OAuth2 authentication using the Authorization Code flow.
Supports Google, GitHub, and Microsoft providers.

Fixes #42
```

### The Prompt

The commit message generation prompt includes the diff inline:

**Location**: `src/prompts/commit.rs` (see `prompt_generate_commit_message_with_diff`)

The function generates a prompt that includes the diff inline and requests a Conventional Commits formatted message.

**Key point**: The diff is passed inline in the prompt - the agent doesn't run git commands.

## Agent Isolation

### What Agents Don't See

AI agents are completely isolated from git operations:

1. **No git commands in prompts**: Agent prompts never contain `git diff`, `git status`, `git commit`, etc.
2. **No git context**: Agents don't know about commits, branches, or git history
3. **No file I/O for git**: Agents don't write `.agent/start_commit` or commit messages
4. **Orchestrator handles git**: Only the orchestrator (Rust code) interacts with git

### Verification

Run the test to verify agent isolation:

```bash
cargo test ralph_agents_dont_receive_git_commands_in_prompts
```

### What Agents Do

Instead of git operations, agents focus on their core tasks:

- **Developer agent**: Reads `PROMPT.md`, creates `PLAN.md`, executes the plan
- **Reviewer agent**: Receives diff inline, creates `ISSUES.md`, reviews code
- **Fix agent**: Reads `ISSUES.md`, applies fixes

The orchestrator handles all file I/O and git operations.

## Cumulative Diffs for Reviewers

### The Problem

Reviewers need to see all changes made during the pipeline, not just per-iteration diffs. This ensures comprehensive review of accumulated work.

### The Solution: `start_commit`

Ralph uses a "start commit" reference to track the baseline:

1. **At pipeline start**: Save current HEAD as `.agent/start_commit`
2. **For each review**: Generate diff from `start_commit` → working tree
3. **Persist across runs**: The `start_commit` file persists until explicitly reset

**Location**: `src/git_helpers/start_commit.rs`

The `save_start_commit()` function saves the current HEAD as the baseline, only if a start commit file doesn't already exist (to persist across runs).

### Incremental Review Mode

When using `--review-depth incremental`, reviewers receive the cumulative diff:

**Location**: `src/phases/review.rs` (see incremental review handling)

The review phase uses `get_git_diff_from_start()` to generate a diff from the start commit baseline to the current working tree, providing reviewers with a comprehensive view of all changes.

### Resetting the Baseline

Users can reset the `start_commit` to establish a new baseline:

```bash
ralph --reset-start-commit
```

This updates `.agent/start_commit` to the current HEAD, useful when:
- Starting a new feature
- After manual commits
- When you want reviewers to see a smaller diff

## Deterministic Operations in Rust

### File I/O

All file I/O is handled by the orchestrator in Rust:

**PLAN.md** (`src/phases/development.rs`):
- Agent returns plan content (captured by JSON parser)
- Orchestrator extracts content from agent output
- Orchestrator writes to `.agent/PLAN.md`
- Fallback: Extract from JSON log if direct write fails

**ISSUES.md** (`src/phases/review.rs`):
- Agent returns issues content (captured by JSON parser)
- Orchestrator extracts content from agent output
- Orchestrator writes to `.agent/ISSUES.md`
- Fallback: Extract from JSON log if direct write fails

### Git Operations

All git operations use libgit2 directly:

**Location**: `src/git_helpers/repo.rs`

- `require_git_repo()` - Check if in a git repository
- `get_repo_root()` - Get repository root path
- `git_snapshot()` - Get status in porcelain format
- `git_diff()` - Get HEAD → working tree diff
- `git_add_all()` - Stage all changes
- `git_commit()` - Create a commit
- `has_meaningful_changes()` - Check for non-whitespace changes

**No git CLI required**: All operations work via libgit2, even if git is not installed.

## Risks and Mitigations

### Risk 1: LLM Failure for Commit Message Generation

**Mitigation**: `commit_with_auto_message()` includes fallback behavior. If the LLM fails, it uses `generate_fallback_commit_message()` to create a descriptive message based on the diff.

### Risk 2: Empty Repository (No Commits)

**Mitigation**: The code in `app/mod.rs` handles this gracefully with a warning and fallback to using HEAD directly. The test `ralph_save_start_commit_handles_empty_repo` verifies this behavior.

### Risk 3: Whitespace-Only Changes

**Mitigation**: The `has_meaningful_changes()` function filters out whitespace-only changes. It checks for actual content changes beyond diff metadata and whitespace lines.

### Risk 4: Agent Output Extraction Failure

**Mitigation**: Both `development.rs` and `review.rs` include fallback logic (`try_extract_plan_from_log` and `try_extract_issues_from_log`) that parse JSON logs to extract content when direct file creation fails.

### Risk 5: Test Coverage Gaps

**Mitigation**: Comprehensive integration tests verify all acceptance criteria without requiring external dependencies.

## Troubleshooting Commit Message Issues

### Problem: Commit Messages Are Too Generic

**Symptoms**: Messages like "chore: 3 files changed" or "chore: apply changes"

**Causes**:
- LLM not analyzing the diff properly
- Diff too large for LLM to process effectively
- LLM timeout causing fallback

**Solutions**:
1. Check for warnings about large diffs:
   ```
   Warning: Large diff detected (X bytes). This may affect commit message quality.
   ```
2. If you see this, consider:
   - Making smaller, more focused changes
   - Breaking up large refactors into smaller commits
3. Check the LLM agent is responding within 60 seconds
4. Verify your LLM agent supports Conventional Commits format

### Problem: Commit Messages Don't Follow Conventional Commits

**Symptoms**: Messages like "Updated the code" or missing type prefix

**Causes**:
- LLM not following the format specification
- Extraction failure returning raw LLM output

**Solutions**:
1. Check for extraction warnings:
   ```
   Warning: LLM output extraction warning (format=X): ...
   ```
2. Verify your LLM agent supports structured output
3. Try a different LLM agent (e.g., Claude vs Codex)

### Problem: Commits Not Being Created

**Symptoms**: No commits appear despite changes being made

**Causes**:
- Only whitespace changes (filtered out)
- No files staged (e.g., only ignored files)
- LLM and fallback both failing

**Solutions**:
1. Check if changes are meaningful (not just whitespace)
2. Verify files aren't in `.gitignore`
3. Look for error messages in the output
4. Check that files aren't internal artifacts (`.agent/*`, `.git/*`)
